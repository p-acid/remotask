# sharpen-spec

`specs/NNN-<name>.md` 한 파일을 입력으로 받아, `[NEEDS CLARIFICATION: ...]` 마커와 검출된 모호 지점을 사용자와의 batch Q&A로 좁힌다. 머지 본에는 마커가 남아있으면 안 된다 (`CONSTITUTION.md` L257) — 이 스킬이 그 책임을 진다.

## 파이프라인 위치

```
[create-spec] → [validate-spec] → [check-constitution] → [sharpen-spec] ← 여기 → 완성
```

전체 그림은 `.claude/skills/spec-pipeline.md` 참조.

## 언제 쓰나

- `create-spec` chain의 마지막 단계로 자동 호출 (가장 흔한 경우).
- 단독 호출: 며칠 전에 작성한 spec을 마저 다듬을 때, 외부 spec을 인계받았을 때, 같은 spec을 여러 번 라운드로 다듬을 때.
- `validate-spec` 또는 `check-constitution`이 검출한 추가 모호 지점을 함께 처리하고 싶을 때.

## 입력

- spec 파일 경로 (예: `specs/008-dm-channel-mode.md`).
- 명시되지 않으면 `specs/`에서 가장 최근에 변경된 파일을 후보로 제시하고 사용자에게 확인.
- 선행 게이트(`validate-spec` / `check-constitution`)가 검출한 추가 ambiguity 항목 목록 (chain 호출 시 컨텍스트로 함께 들어옴).

## 워크플로우

### 1. 모호 지점 수집

다음을 모두 모은다:

- **명시 마커**: `[NEEDS CLARIFICATION: ...]` 텍스트 grep.
- **선행 게이트 보고**: validate-spec / check-constitution이 PASS 하지 않았거나 "검사 통과했지만 모호하다"고 표시한 항목.
- **자체 검출**: 다음 패턴은 마커가 없어도 모호로 간주.
  - AT가 goal-form ("make X work", "system handles Y" 같은 관찰 불가능 문구).
  - Behavior에 구현 detail이 새어든 부분 (함수 시그니처, SQL 컬럼명).
  - Constitution check 슬롯이 "PASS — 영향 없음"만 적혀 있고 *왜 영향 없는지*가 비어 있는 경우.
  - waiver가 3줄 형식 (위반 / 왜 / 단순 대안 거부)을 채우지 않은 경우.
  - Notes에 "D24 candidate" 표기는 있는데 한 줄 요약이 빈 경우.

이 단계에서 모호 지점이 0개면 사용자에게 "마커 / 모호 지점 없음. spec 그대로 머지 가능한 상태입니다"라고 알리고 끝낸다.

### 2. Batch Q&A

수집된 항목을 **한 번에** 사용자에게 묻는다. 한 항목씩 따로 묻지 마라 — 왕복 횟수가 사용자 시간을 가장 많이 잡아먹는다. 각 항목은 정확히 네 줄로:

```
N) <어디에 있는지>: <무엇이 모호한가, 한 줄>
   - 옵션 A — <짧은 설명>. 트레이드오프: <한 줄>.
   - 옵션 B — <짧은 설명>. 트레이드오프: <한 줄>.
   - 추천: <A 또는 B 또는 별도 안>. 이유: <단순함 / MVP-First / 기존 ARD 정합성 같은 한 가지 기준>.
```

옵션은 2~3개. 추천은 반드시 명시 — `CLAUDE.md` §1 "다중 해석이 있으면 모두 제시하고 침묵으로 고르지 마라"에 따른다. 또한 같은 §1의 "더 단순한 접근이 있으면 말하고 정당하면 반박하라" — 추천이 사용자가 던진 원래 방향과 다르면 그 차이를 한 줄로 설명한다.

자주 발화하는 모호 후보:

- **Behavior 경계** — 어디까지 이 spec에 포함할지. 더 큰 변경이면 spec을 쪼개는 게 맞고, "이 spec은 X까지만, Y는 후속 spec"으로 Out-of-scope에 명시.
- **Phase 부합성** — 이 변경이 현재 Phase 안인지, 미래 Phase를 미리 당기는 건지. 후자면 §IV waiver의 무게와 "현재 Phase에 머무르기" 단순 대안을 함께 제시.
- **새 ARD 필요 여부** — 시스템 shape이 바뀌면 동행 ARD entry를 만들지. 만든다면 D번호 후보 + 한 줄 요약.
- **AT 관찰 가능 outcome** — "test passes" 같은 모호한 후미 대신 "stdout에 `STEP <body>` 라인이 N번 emit된다"처럼 검증 지점이 분명한 표현으로.
- **waiver 단순 대안** — waiver의 (c)줄을 채울 단순 대안 후보 (보통 "현재 상태 유지").

### 3. 답을 받아 spec 파일 in-place 수정

사용자 답을 받으면 spec 파일을 수정한다 — 새 파일을 만들지 말고 같은 경로를 in-place 편집:

- `[NEEDS CLARIFICATION: <질문>]` 마커는 사용자 답으로 그 자리를 치환.
- 자체 검출한 항목은 사용자 답을 반영해 해당 문장 / 섹션을 다시 쓴다.
- AT goal-form이었다면 Given/When/Then 형태로 재작성.
- Constitution check 한 줄짜리 슬롯에 *왜 영향 없는지* 한 단어를 보강 (예: "PASS — 영향 없음" → "PASS — D19가 channel mapping을 presentation layer로 이미 옮김").

수정 후 변경된 줄들을 짧게 사용자에게 보고. 한 라운드에 보통 3~7줄 정도 변한다.

### 4. 추가 라운드 / 종료

수정된 spec에서 다시 모호 지점이 발견되면 (사용자 답이 새 모호함을 만들었거나, 한 라운드 만에 못 잡힌 항목) 2단계로 돌아간다. 보통 1~2 라운드면 닫힌다.

사용자가 "충분해" / "좋아"라고 하거나, 1단계 수집에서 모호 지점이 0개가 되면 종료. 종료 시:

- 마지막 grep — `[NEEDS CLARIFICATION` 가 spec 파일에 0개 남아있는지 검증.
- 한 줄로 사용자에게 알린다: "마커 0개 / 모호 지점 정리 완료. spec 머지 게이트 통과 가능."

## 흔한 실수

- **한 항목씩 따로 묻기.** 왕복 5번 ↔ 1번의 차이가 사용자 시간을 결정한다. 모은 다음 한 번에.
- **추천 없이 옵션만 나열.** §1 원칙을 어기는 짓. 추천 + 이유를 반드시 적는다. 추천이 없으면 사용자가 "어느 게 나아?"를 다시 물어 한 라운드를 더 쓴다.
- **답을 받았는데 spec 파일을 안 고친다.** 이 스킬의 진짜 산출물은 *수정된 spec 파일*이다. 사용자에게 답만 정리해서 보여주고 끝내지 마라.
- **자체 검출 누락.** 마커만 보고 끝내면 goal-form AT, 비어있는 Constitution check 슬롯 같은 것은 못 잡는다. 1단계의 자체 검출 패턴 목록을 매번 한 번 훑어라.
- **마커 잔존 검증 누락.** 종료 직전에 grep으로 한 번 더 확인. 이게 머지 게이트의 가장 단순한 빨간불이다.

## 작성 톤

- 사용자와의 Q&A는 **한국어**로.
- spec 파일에 쓰는 본문은 **영어**로 (durable docs 컨벤션, ARD D23).
- 마커 치환 시 원문 톤을 유지 — 차분한 영문체, 마케팅 문구·강한 형용사 회피.
